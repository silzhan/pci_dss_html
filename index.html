02定位与边界
PCI DSS 是什么
支付责任边界，不是通用安全清单
核心：卡数据流 + CDE 安全影响
谁处理卡、数据流经哪里、谁影响 CDE
200 个微服务，可能只有少数直接处理卡数据
本质是架构决策，不是安全团队的活
Hosted Page vs Direct API → 未来三年合规成本
v4.0：从"年度大考"变成"持续运营"
日志、告警、变更管理都得自动化
常见误解 ❌
✗"没有直接收卡号就没事"
→ ELK 一条日志打 PAN，整个集群进 CDE
✗"PCI 是安全团队的事"
→ 产品选型那一刻，合规成本就定了
✗"年底突击三个月能过"
→ v4.0 要求持续监控、持续评估
✗"PCI = ISO27001 通用安全清单"
→ 关注卡数据生命周期和 CDE 安全影响

09密钥管理 · 分层加密模型

← 用 DEK 加密
↓
🔑 DEK (Data Encryption Key)
加密 PAN 的密钥，存在 DB（被 KEK 包过）
← 用 KEK 加密
↓
🔐 KEK (Key Encryption Key)
在 KMS/HSM 中，密钥与密文分离
↓

11数据治理 · 卡数据副本追踪
💳 CDE 卡数据
受控存储点
📝 ELK / 日志
错误日志打了完整 PAN
→ 整个集群进 CDE
📊 APM / Trace
采了请求体含卡号
→ APM 进 CDE
🎧 客服后台
截图里有卡号
→ 客服系统进 CDE
📈 BI / 数仓
同步了未脱敏数据
→ 数仓进 CDE
🧪 测试环境
用了真实卡号
→ 重大不符合风险
治理规则 · v4.0 要求
日志：全链路不记录 PAN/CVV，正则脱敏，权限 + 留存周期
客服：只展示后四位 + 卡品牌，全量审计，权限拆分
BI：PAN → token/hash，CVV 不进，BIN 可进
AI 风控：用 BIN/token/PSP 风控分/合成数据
